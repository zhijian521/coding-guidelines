# 二、命名规范

命名的核心就一条：**让读代码的人不需要查注释就知道这个变量是干什么的**。名称长一点没关系，现代编辑器都有自动补全。名称短但让人猜才是问题。

## 2.1 项目与目录

| 类别 | 规则 | 示例 |
|---|---|---|
| 项目名 | 全小写 + 中划线分隔 | `my-project` |
| 目录名 | 全小写 + 中划线，复数用复数 | `components`、`utils` |
| 组件目录 | kebab-case | `user-profile` |

为什么统一小写+中划线？因为 Git 在大小写不敏感的系统上（比如 macOS 默认）容易出问题，中划线比下划线更易读，而且 npm 包名也是这个规范。

## 2.2 文件

| 类型 | 规则 | 示例 |
|---|---|---|
| JS / TS 工具 | kebab-case | `date-formatter.ts` |
| Vue 组件 | PascalCase 或 kebab-case | `UserCard.vue` |
| React 组件 | PascalCase，`.tsx` | `UserCard.tsx` |
| CSS / SCSS | kebab-case | `user-card.module.scss` |

注意：组件文件和组件名保持一致。如果一个组件叫 `UserCard`，文件名必须是 `UserCard.vue` 或 `UserCard.tsx`。例外是 Vue 项目里用 `index.vue` 作为目录默认入口的情况，但组件外的引用路径仍是 PascalCase 的目录名。

## 2.3 变量与函数

```typescript
// 变量：小驼峰 (lowerCamelCase)
const userName = 'Alice';
const orderList = [];

// 常量：全大写 + 下划线
const MAX_SIZE = 100;
const API_BASE_URL = 'https://api.example.com';

// 函数：动词 + 名词
function getUserInfo(userId: string) {}
function handleSubmit() {}

// 布尔值：is / has / can 前缀
const isLoading = true;
const hasPermission = false;

// 类 / 接口 / 类型：大驼峰 (UpperCamelCase)
class UserController {}
interface UserProfile {}
type OrderStatus = 'pending' | 'completed';

// 私有属性：下划线前缀
class Store {
  private _cache: Map<string, unknown>;
}
```

## 2.4 命名对比

**不推荐：**
```javascript
// 无意义名称
const d = new Date();
function fn() {}

// 拼音
const yonghuMing = '';

// 魔法数字
if (status === 1) {}
```

**推荐：**
```javascript
// 语义化
const currentDate = new Date();
function calculateTotal() {}

// 英文
const userName = '';

// 语义化常量
if (status === OrderStatus.Pending) {}
```

## 2.5 事件与路由命名

| 场景 | 规则 | 示例 |
|---|---|---|
| 事件处理函数 | `handle` + 事件名 | `handleClick`、`handleSubmit` |
| 事件回调 prop | `on` + 事件名 | `onClick`、`onChange` |
| 路由 path | kebab-case | `/user-profile`、`/order-detail` |
| 路由 name | camelCase | `userProfile`、`orderDetail` |
| 枚举 key | PascalCase | `OrderStatus.Pending` |
| 枚举 value | UPPER_SNAKE_CASE | `'PENDING'`、`'SHIPPED'` |

## 2.6 缩写词处理

业界常见的缩写词（如 URL、API、XML、JSON）在命名中保持原样：`URLParser`、`XMLHttpRequest`、`parseJSON`（小驼峰中后置则首字母大写）。自己发明的缩写一律不可以用——`usrNm`、`btnClk` 这种写法省不了几个字符，但读起来要花几秒解码。

> **团队共识：** 不要用拼音，也不要用自造缩写（比如 `usrNm`）。新同事加入时看代码应该是自然阅读的感觉，不是在解密。
