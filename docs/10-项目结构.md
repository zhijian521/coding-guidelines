# 十、项目结构

目录结构的目标是**让新加入的同事能在 10 分钟内找到他需要的文件**。不是追求完美分类，而是追求直觉可预测——看到一个文件名，不用想就知道它应该在哪个目录。

## 10.1 标准目录

```
project/
├── public/              静态资源（不参与打包）
├── src/
│   ├── api/             接口封装
│   │   ├── modules/     按业务模块拆分
│   │   └── request.ts   axios 实例
│   ├── assets/          静态资源（参与打包）
│   ├── components/      公共组件
│   │   ├── base/        基础 UI 组件
│   │   └── business/    业务组件
│   ├── composables/     组合式函数 / Hooks
│   ├── constants/       常量定义
│   ├── pages/           页面组件
│   ├── router/          路由配置
│   ├── store/           状态管理
│   ├── styles/          全局样式
│   ├── types/           TypeScript 类型
│   └── utils/           工具函数
├── .env                 环境变量
├── .eslintrc.cjs
├── .prettierrc
├── tsconfig.json
├── vite.config.ts
└── package.json
```

## 10.2 目录职责

| 目录 | 职责 | 约束 |
|---|---|---|
| `api/` | 接口定义与请求封装 | 按模块拆分，统一错误处理 |
| `components/` | 可复用组件 | 禁止包含业务逻辑 |
| `pages/` | 页面级组件 | 每个页面对应一个路由 |
| `composables/` | 跨页面复用的组合逻辑 | 以 `use` 开头命名 |
| `router/` | 路由定义与守卫 | 使用懒加载 |
| `store/` | 全局状态管理 | 按模块拆分 |
| `styles/` | 全局样式、变量 | 不放组件级样式 |
| `types/` | 公共类型定义 | 与业务类型分离 |
| `utils/` | 纯工具函数 | 无副作用 |

## 10.3 按功能组织（Feature-based）

对于业务复杂的项目，推荐按功能模块而非文件类型组织代码。每个 feature 目录内包含该功能的所有相关文件。

```
src/features/
├── user/
│   ├── components/     UserCard.tsx, UserList.tsx
│   ├── hooks/          useUserList.ts
│   ├── api/            userApi.ts
│   └── types/          user.ts
├── order/
│   ├── components/     OrderTable.tsx
│   ├── hooks/          useOrderFilter.ts
│   └── api/            orderApi.ts
└── shared/             跨 feature 的公共代码
```

> 页面路由与文件路径保持对应：`/pages/user/profile.tsx` → `/user/profile`。这样看到 URL 就知道去哪个文件改代码。
