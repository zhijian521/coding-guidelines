# 七、注释规范

注释的作用是解释**为什么**这样做，而不是**做了什么**——后者代码本身已经说清楚了。好的注释让接手的人不用猜你的意图，坏的注释只是在制造噪音。

## 7.1 单行注释

用 `//` 后跟一个空格。写注释前先问自己：这段代码删掉注释后，下一个人还能不能看懂？如果能，说明注释在重复代码，可以删。

```javascript
// 注释前保留一个空格

// 计算税费：税率根据地区动态获取 (API: /tax/rate)
const tax = amount * getTaxRate(region);

// ❌ 不推荐 — 注释重复代码
// 设置 name 为 'Alice'
const name = 'Alice';
```

## 7.2 JSDoc 注释

函数、类、接口必须有 JSDoc 注释。不需要每行都写，但要覆盖参数含义、返回值、可能抛出的异常。编辑器会根据 JSDoc 提供智能提示，调用方不用跳转到定义就能知道怎么用。

```typescript
/**
 * 根据用户 ID 获取用户信息
 * @param userId - 用户唯一标识
 * @param options - 可选配置
 * @param options.includePosts - 是否包含文章列表
 * @returns 用户信息对象，失败时返回 null
 * @throws {ApiError} 网络请求失败时抛出
 */
async function fetchUser(
  userId: string,
  options?: { includePosts?: boolean }
): Promise<User | null> {
  // ...
}

/**
 * 订单状态枚举
 */
enum OrderStatus {
  /** 待支付 */
  Pending = 'PENDING',
  /** 已发货 */
  Shipped = 'SHIPPED',
}
```

## 7.3 TODO / FIXME 标记

这些标记让团队能快速定位需要关注的代码。IDE 通常会自动高亮它们。标记后最好带上 JIRA 编号或负责人。

| 标记 | 用途 | 示例 |
|---|---|---|
| `TODO` | 待完成的功能或优化 | `// TODO: 接入真实支付接口 (JIRA-456)` |
| `FIXME` | 已知问题，需要修复 | `// FIXME: 浮点数精度问题，改用 decimal.js` |
| `HACK` | 临时方案，需要重构 | `// HACK: 绕过 IE 兼容问题，下个版本移除` |
| `NOTE` | 重要说明信息 | `// NOTE: 此接口每分钟限流 100 次` |

## 7.4 CSS 注释规范

```css
/* ===== 页面头部 ===== */
.header {
  display: flex;
}

/* ===== 商品卡片 ===== */
.product-card { }
.product-card__image { }
.product-card__price { }
```

## 7.5 HTML 注释规范

```html
<!-- 页面头部 -->
<header>
  <nav>...</nav>
</header>

<!-- 主要内容区 -->
<main>
  <!-- TODO: 待添加侧边栏 -->
</main>
```

> 注释掉的旧代码不得长期保留。提交前应删除所有调试用的 `console.log`。

## 7.6 文件头注释

每个源文件顶部应有简短描述。

```typescript
/**
 * 日期格式化工具函数
 * 提供项目中统一的日期展示、计算和校验方法
 */
export function formatDate(date: Date): string { /* ... */ }

/** 用户信息卡片，支持编辑和删除两种模式 */
export const UserCard = () => { /* ... */ }
```

## 7.7 @deprecated 标记

废弃的 API、函数、组件用 `@deprecated` 标记，同时写明替代方案。给调用方一个迁移窗口期。

```typescript
/**
 * @deprecated 自 v2.3 起废弃，请使用 {@link fetchUserV2} 代替。
 * 该函数将在 v3.0 中移除。
 */
export function fetchUser(id: string) { /* ... */ }
```
